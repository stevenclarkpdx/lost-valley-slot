Lost Valley Prototype Architecture

Goal
Build a browser-based slot-design sandbox whose base game feels like a
prehistoric expedition and whose feature system can support multiple themed
valleys without duplicating math or presentation flow.

Current game
- Base game: 5x5 independently weighted symbol board.
- Payouts: orthogonal cluster pays for 4+ matching cluster symbols.
- Field Notes: unique natural Evidence symbols award escalating instant bonuses.
- Destination triggers:
  - 3+ Fossil Footprints trigger Fossil Valley.
  - 3+ Predator Tracks trigger Predator Valley.
  - 4 trigger symbols add +1 starting respin; 5+ add +2.
- Wilds substitute for cluster-paying symbols only.
- Feature sessions are 5x5 reveal boards with respins, hit resets, misses, tile
  payouts, and optional generic progression.

Engine boundaries
- src/engine is pure TypeScript and has no React/browser dependencies.
- All randomness consumes injected seeded RNG instances.
- GameConfig.featureProfiles is the authoritative multi-feature configuration.
- Legacy imported featureProfile JSON is normalized into featureProfiles.
- FeatureEngine is profile-driven and contains no Fossil- or Predator-specific
  logic.
- Fossil Assembly and Tracking Confidence are presentation/profile concepts over
  the generic progression contract.

Presentation boundaries
- React owns animation, layout, credit display, and thematic language.
- Engine results are resolved before presentation sequences consume them.
- Presentation beats are data-shaped phase/duration/action records scheduled by
  src/presentation/sequenceRunner.ts.
- Skip Anim and reduced-motion paths should compress timing without changing
  engine results.

Analytics
Simulation uses the production engine and reports:
- Base RTP
- Evidence RTP
- Combined feature RTP
- Total RTP
- Combined trigger frequency and feature win percentiles
- Per-feature trigger frequency, average win, RTP contribution, percentiles,
  final reveal distributions, and full reveal rate
- Footprint and Predator Track count distributions
- Cluster, symbol, wild, Golden Amber, and Field Notes diagnostics

Engineering priorities
- Keep math centralized and deterministic.
- Keep feature profiles data-first.
- Keep presentation state unable to mutate gameplay state.
- Prefer small extraction seams that make iteration faster without broad visual
  rewrites.
- Do not commit generated dist/cache files.

Known remaining technical debt
- App.tsx still carries more high-churn UI code than ideal. Symbol/profile
  resolution and sequencing are now extracted, but BaseGame, FeatureBoard,
  FieldNotesPanel, CreditPanel, SimulationPanel, and Diagnostics should continue
  moving into focused components during a future low-risk UI extraction pass.
